# Free Roam Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** An opt-in "Free roam" setting that makes every skill openable regardless of prerequisites, per `docs/superpowers/specs/2026-07-20-free-roam-design.md`.

**Architecture:** A persisted optional boolean `unlockAll` on `ProgressState` is OR-ed into the app's single unlock derivation (`HomeScreen.tsx:88`). This is a gate bypass — nothing ever writes `completed: true`, so the feature is fully reversible and leaves Daily Review, badges, XP, and catches untouched. Not-yet-earned nodes stay visually marked via a new `.node.locked` CSS rule (needed because their current dimmed look comes only from `button:disabled`, which free roam removes).

**Tech Stack:** React + zustand + idb-keyval, vitest, Playwright.

**Critical trap (from the spec):** `ProgressState` is re-materialized from a hardcoded field list in **three** functions — `dataOf` (`progress.ts:89`), `importState` (`:195`), `exportState` (`:210`). A field added to the interface but missed in `dataOf` works until the user's next solve/catch/badge/review and then silently reverts. Task 1's test suite is designed to catch exactly that.

```ts
test('setUnlockAll survives a later mutation', () => {
  useProgress.getState().setUnlockAll(true)
  expect(useProgress.getState().unlockAll).toBe(true)
  useProgress.getState().recordSolve('select-basics', 'sb-1', 10, 0, 2)
  expect(useProgress.getState().unlockAll).toBe(true)
})

test('unlockAll round-trips through export and import', () => {
  useProgress.getState().setUnlockAll(true)
  const json = exportState(useProgress.getState())
  expect(JSON.parse(json).unlockAll).toBe(true)
  useProgress.getState().setUnlockAll(false)
  useProgress.getState().importState(JSON.parse(json) as ProgressState)
  expect(useProgress.getState().unlockAll).toBe(true)
})

test('a save without unlockAll hydrates falsy and is not rewritten', async () => {
  const { set: idbSet, get: idbGet } = await import('idb-keyval')
  const saved = {
    version: 1,
    xp: 30,
    streak: { count: 1, lastDay: '2026-07-19' },
    skills: { 'select-basics': { solved: ['sb-1'], completed: false, mastery: 0 } },
    collection: [],
    badges: [],
  }
  await idbSet('sql-quest-progress', saved)
  await useProgress.getState().hydrate()
  expect(useProgress.getState().unlockAll).toBe(false)
  expect(await idbGet('sql-quest-progress')).toEqual(saved)
})
```

Note the third test's purpose: `hydrate` re-persists whenever the normalized state differs from what was saved, so it fails if `normalize()` defaults the field.

- [ ] **Step 5: Thread the field through all three serialization sites.**

`dataOf` (line 89):

```ts
function dataOf(s: ProgressStore): ProgressState {
  return {
    version: 1,
    xp: s.xp,
    streak: s.streak,
    skills: s.skills,
    collection: s.collection,
    badges: s.badges,
    unlockAll: s.unlockAll,
  }
}
```

`importState` (line 195):

```ts
  importState(imported) {
    if (!isProgressState(imported)) throw new Error('Unrecognized progress file')
    const next = normalize({
      version: 1,
      xp: imported.xp,
      streak: imported.streak,
      skills: imported.skills,
      collection: imported.collection,
      badges: imported.badges,
      unlockAll: imported.unlockAll,
    })
    set(next)
    persist(next)
  },
```

`exportState` (line 210):

```ts
export function exportState(s: ProgressState): string {
  return JSON.stringify(
    {
      version: s.version,
      xp: s.xp,
      streak: s.streak,
      skills: s.skills,
      collection: s.collection,
      badges: s.badges,
      unlockAll: s.unlockAll,
    },
    null,
    2,
  )
}
```

**Do NOT touch `normalize()`** — `hydrate` re-persists whenever `JSON.stringify(normalize(saved)) !== JSON.stringify(saved)`, so a default there would rewrite every existing save on load (the Step 1 third test guards this).

**DO default it in `empty`**, so runtime state is always a real boolean and no consumer needs `?? false`. `empty` is not part of the rewrite comparison (verified: `hydrate` compares normalized-vs-saved only), so this is safe. In the `empty` constant add `unlockAll: false`, and in `importState` coerce the incoming value so a legacy file without the field clears the flag to `false` rather than `undefined`:

```ts
      unlockAll: imported.unlockAll ?? false,
```

The persisted type stays optional (`unlockAll?: boolean`) so old saves and old export files remain valid.

- [ ] **Step 6: Implement the action.** Add to the store object, next to `importState`:

```ts
  setUnlockAll(value) {
    const next: ProgressState = { ...dataOf(get()), unlockAll: value }
    set(next)
    persist(next)
  },
```

If the store factory's signature does not already expose `get`, use the same accessor pattern the neighboring actions use (several read state via `useProgress.getState()`); match whichever is already present rather than changing the factory signature.

Use this final version — a locked-and-unreachable node keeps 🔒; a roamed-into node shows 🔓.

- [ ] **Step 1: Add the e2e test.** Append to `e2e/smoke.spec.ts`, following the file's existing `addInitScript` seeding pattern:

```ts
test('free roam opens a skill whose prerequisites are unmet', async ({ page }) => {
  await page.addInitScript(() => {
    const req = indexedDB.open('keyval-store')
    req.onupgradeneeded = () => req.result.createObjectStore('keyval')
    req.onsuccess = () => {
      const tx = req.result.transaction('keyval', 'readwrite')
      tx.objectStore('keyval').put(
        {
          version: 1,
          xp: 0,
          streak: { count: 0, lastDay: '' },
          skills: {},
          collection: [],
          badges: [],
          unlockAll: true,
        },
        'sql-quest-progress',
      )
    }
  })
  await page.goto('/')
  await page.getByRole('button', { name: /Free roam: on/ }).click({ trial: true })
  const roamed = page.getByRole('button', { name: /Recursive CTEs/ })
  await expect(roamed).toHaveClass(/locked/)
  await expect(roamed).toContainText('🔓')
  await roamed.click()
  await expect(page.getByRole('heading', { name: 'Recursive CTEs' })).toBeVisible()
  await expect(page.getByRole('button', { name: 'Start exercises' })).toBeVisible()
})
```

The `locked` class and 🔓 assertions are load-bearing: they pin the `earned`-vs-`unlocked` asymmetry. Without them, someone could collapse `earned` into `unlocked` in the className and badge expressions and every test would still pass while the tree silently stopped showing real progress.

`recursive-cte` requires `cte`, which the seeded save has not completed, so this fails without the bypass. The `trial: true` click asserts the toggle is present and enabled without actually flipping it.
